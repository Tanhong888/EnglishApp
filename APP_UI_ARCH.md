# 前端界面开发文档

## 1. 文档目标

本文档用于指导 `desktop_app` 前端界面的设计与开发，目标是让 EnglishAPP 的桌面端界面保持以下特征：

- 美观简约，不过度装饰
- 符合大众审美，视觉稳定耐看
- 信息层级清晰，易学易用
- 便于 Flutter 组件复用与后续迭代

本文档以仓库当前代码为基础，适用于以下主要页面：

- 启动页
- 登录页
- 首页
- 文章库页
- 阅读详情页
- 生词本页
- 生词详情页
- 我的页
- 设置页
- 内容运营页

---

## 2. 设计关键词

本项目的界面风格建议统一为以下关键词：

- 清爽：以浅色背景为主，减少压迫感
- 简洁：避免复杂纹理、重阴影、大面积高饱和色
- 易读：优先保证文字信息的可读性和信息密度平衡
- 克制：强调重点，不堆叠视觉元素
- 温和：按钮、卡片、输入框采用中小圆角和柔和对比

建议整体气质参考「教育工具 + 内容产品」的结合风格，而不是强营销感或强科技感。

---

## 3. 视觉基调

### 3.1 色彩策略

建议使用“中性色背景 + 蓝色品牌强调 + 成功/警告/错误辅助色”的方案。

当前项目已存在的主题色可作为基础：

- 背景色：`#F7F8FA`
- 卡片色：`#FFFFFF`
- 主文字：`#111827`
- 次文字：`#6B7280`
- 品牌色：`#2563EB`
- 品牌浅色：`#E8F0FF`
- 边框色：`#E5E7EB`
- 成功色：`#16A34A`
- 警告色：`#D97706`
- 错误色：`#DC2626`

使用原则：

- 页面主背景使用浅灰白，避免纯白刺眼
- 主要操作按钮使用品牌蓝
- 文字始终优先保证对比度
- 标签、筛选、状态色只做点缀，不大面积铺色
- 同一页面内高饱和颜色不超过 2 种

### 3.2 字体策略

当前项目已使用 `GoogleFonts.notoSansTextTheme`，建议继续沿用。

建议规则：

- 中文正文使用 Noto Sans 系列，风格中性稳定
- 标题通过字重和字号建立层级，不依赖花哨字体
- 英文单词、音标、标签类内容可适当提高字重，但不要跳脱整体风格

推荐字号层级：

- 页面主标题：28
- 区块标题：20
- 卡片标题：16 到 18
- 正文：14 到 16
- 辅助信息：12 到 13
- 标签文字：12

### 3.3 圆角与阴影

建议整体采用轻圆角风格：

- 输入框圆角：10
- 卡片圆角：12
- 弹层圆角：16
- 胶囊标签圆角：999

阴影使用原则：

- 默认弱阴影或无阴影
- 更推荐使用边框、背景层级和留白区分区域
- 仅在弹层、悬停卡片等少数场景使用轻微阴影

## 4. 版式规范

### 4.1 页面结构

桌面端页面建议统一为三段结构：

1. 顶部导航区
2. 主内容区
3. 底部导航区

当前项目的 `Scaffold + AppBar + NavigationBar` 模式可以继续保持。

### 4.2 内容宽度

桌面端虽然空间更大，但不建议让正文无限拉宽。建议：

- 登录页主体宽度控制在 `420` 到 `480`
- 列表页内容宽度以舒适阅读为准，建议主列最大宽度 `960` 到 `1120`
- 文章详情正文列建议控制在 `760` 到 `900`
- 超宽屏场景优先增加留白，不要盲目拉伸内容

### 4.3 间距系统

当前项目已定义：

- `xs = 8`
- `sm = 12`
- `md = 16`
- `lg = 20`
- `xl = 24`

建议统一规则：

- 同组元素内部间距优先使用 `8` 或 `12`
- 模块之间间距优先使用 `16` 或 `20`
- 大区块分隔使用 `24`
- 页面左右内边距保持 `16` 起步，大屏可扩展到 `24`

### 4.4 对齐原则

- 所有页面优先左对齐
- 标题、说明文、操作按钮按同一基线组织
- 列表卡片中的标题、元信息、操作按钮保持固定顺序
- 避免一个页面同时出现过多居中布局，居中只用于登录、空态、加载态

## 5. 信息架构建议

### 5.1 一级导航

当前底部导航结构合理，建议继续保持 3 个一级入口：

- 首页
- 生词本
- 我的

原因：

- 首页承担内容发现与任务开始
- 生词本承担高频复习与检索
- 我的承担账号、数据、设置和后台入口

### 5.2 二级页面关系

建议页面关系如下：

- 首页
  - 文章库
  - 阅读详情
- 生词本
  - 生词详情
- 我的
  - 行为详情
  - 设置
  - 内容运营
  - 文章编辑

### 5.3 页面层级原则

- 一级页面只承载核心任务，不放太多解释文字
- 二级页面重点解决单一任务
- 详情页强调沉浸与效率
- 后台页强调清晰和可靠，不追求花哨

## 6. 组件设计规范

### 6.1 按钮

建议统一三类按钮：

- `FilledButton`：主操作，如登录、进入文章、提交
- `OutlinedButton`：次操作，如取消、查看更多、切换状态
- `TextButton`：弱操作，如跳转详情、辅助链接

规则：

- 一个区块内主按钮最多 1 个
- 避免同一行出现多个同权重主按钮
- 按钮文案使用动词开头，例如“登录并进入首页”“查看详情”“加载更多”

### 6.2 卡片

卡片是当前项目的核心容器，建议继续作为主信息承载组件。

适用场景：

- 首页推荐
- 文章列表
- 生词列表
- 用户统计
- 功能入口

卡片规范：

- 内边距建议 `16`
- 标题在上，说明在下，操作在底部或右侧
- 卡片内不要堆叠过多边框
- 单卡片只突出一个主信息点

### 6.3 表单

表单风格应保持克制、稳定、易扫读。

规范：

- 输入框统一使用填充式浅背景
- 标签文案简短明确
- 出错时在字段附近或按钮上方给出提示
- 登录等关键表单优先使用单列布局

### 6.4 筛选与标签

当前 `ChoiceChip` 的用法适合文章分级筛选，建议规范如下：

- 筛选维度不超过 5 个
- 默认态低对比，选中态使用品牌浅色或品牌色描边
- 标签文案简短，例如“全部”“CET4”“考研”

### 6.5 列表项

列表信息推荐使用统一结构：

1. 标题
2. 元信息
3. 摘要或说明
4. 状态
5. 操作

这样可以保证不同页面之间的认知一致性。

### 6.6 弹层与底部面板

阅读详情中的查词弹层是高频交互，建议遵循：

- 结构短小，优先展示核心信息
- 标题显示单词本体
- 依次展示音标、词性、释义、来源
- 主操作放在底部
- 避免在弹层中再嵌套复杂操作流程

### 6.7 提示反馈

建议统一反馈方式：

- 加载中：`CircularProgressIndicator`
- 成功反馈：`SnackBar`
- 失败反馈：`SnackBar` 或页面内错误提示
- 空态：使用卡片或居中文案，附一条明确下一步建议

## 7. 页面开发规范

### 7.1 启动页

目标：

- 展示品牌名与应用加载状态
- 保持轻量，不停留过久

建议：

- 使用居中 Logo 或应用名
- 搭配简洁进度提示
- 背景保持干净，不建议大面积插画

### 7.2 登录页

目标：

- 低学习成本
- 高识别性
- 聚焦单一任务

建议：

- 表单卡片居中
- 卡片宽度控制在 `420` 左右
- 标题、说明、输入框、按钮垂直排列
- 错误提示贴近操作区域

### 7.3 首页

目标：

- 快速开始学习
- 展示推荐内容
- 承接继续阅读

推荐结构：

1. 欢迎区
2. 快捷操作区
3. 今日推荐
4. 继续阅读

建议：

- 欢迎卡片信息简洁，不写长段产品介绍
- 快捷操作控制在 2 到 3 个
- 推荐文章卡片优先突出标题、级别、主题、时长
- 继续阅读区域强调进度恢复

### 7.4 文章库页

目标：

- 高效率浏览和筛选

推荐结构：

1. 分级筛选
2. 排序方式
3. 文章列表

建议：

- 筛选区固定放在列表顶部
- 排序控件使用单选下拉即可
- 列表卡片信息不宜过多，摘要最多 2 到 3 行

### 7.5 阅读详情页

目标：

- 以阅读为核心
- 弱化外围装饰
- 强化查词和进度记录

建议：

- 顶部先展示标题和文章元信息
- 正文排版要重视行高与段落留白
- 单词点击态要有明显反馈
- 段落容器可使用浅色底增强可点击性
- 收藏操作置于右上角图标位即可

正文排版建议：

- 正文字号 `16`
- 行高 `1.7` 到 `1.9`
- 段落间距 `12` 到 `16`
- 每段使用独立容器，便于点击记录进度

### 7.6 生词本页

目标：

- 快速检索
- 快速识别掌握状态
- 快速进入复习

建议：

- 搜索框固定在顶部
- 状态标签保持短文案，例如“已掌握”“待复习”
- 单词、音标、词性、释义按固定顺序展示
- “查看详情”和“标记掌握”作为主次两个动作

### 7.7 我的页

目标：

- 清楚展示个人状态
- 整合常用功能入口

推荐结构：

1. 用户信息卡
2. 数据概览卡
3. 功能入口列表

建议：

- 统计信息保持摘要化，不堆数字
- 功能入口使用 `ListTile` 风格最稳妥
- 后台类入口放在靠后位置，避免干扰普通用户

### 7.8 设置页与后台页

目标：

- 强调清晰、可靠、低误操作

建议：

- 设置页使用分组列表
- 后台页使用“筛选 + 列表 + 编辑”的经典结构
- 高风险操作使用二次确认

---

## 8. 状态设计规范

每个页面都应明确处理以下 4 类状态：

- 初始加载
- 加载失败
- 空数据
- 正常展示

开发要求：

- 不允许只有正常态，没有空态和错误态
- 异步页面统一支持下拉刷新或显式重试
- 错误文案要让用户知道下一步能做什么

推荐文案风格：

- 空态：直接、简短、带建议
- 错误态：说明原因范围，不暴露过多技术细节
- 成功态：一句话确认结果即可

示例：

- “暂时还没有推荐文章”
- “没有找到匹配结果，可以换个关键词试试”
- “加载失败，请稍后重试”
- “已加入生词本”

## 9. Flutter 落地规范

### 9.1 主题集中管理

界面样式必须优先沉淀到以下位置统一管理：

- `desktop_app/lib/core/theme/tokens.dart`
- `desktop_app/lib/core/theme/app_theme.dart`

要求：

- 颜色、间距、圆角优先抽离为 token
- 不要在页面里频繁硬编码颜色值
- 页面只负责布局和业务表达，不负责定义全局视觉规范

### 9.2 组件复用

当同类结构在 2 个及以上页面重复出现时，建议抽成共享组件。

优先抽离对象：

- 页面内容容器
- 文章卡片
- 状态标签
- 空态区块
- 错误提示区块
- 通用统计卡片

共享组件建议放置于：

- `desktop_app/lib/shared/widgets/`

### 9.3 页面实现原则

- 一个页面只处理一个主任务
- 请求加载逻辑集中在页面状态类或对应 controller
- 页面中的样式写法保持统一，不混用过多不同风格
- 先保证结构清楚，再考虑装饰优化

### 9.4 响应式适配

虽然当前主要是 Windows 桌面端，但仍建议具备基础适配能力。

建议断点：

- 小宽度：小于 `720`
- 中宽度：`720` 到 `1200`
- 大宽度：大于 `1200`

适配原则：

- 小宽度时优先单列
- 大宽度时增加留白，不堆成多栏信息墙
- 正文阅读区始终保持舒适宽度

## 10. 推荐开发流程

建议每个新页面按以下顺序开发：

1. 先确定页面目标和主任务
2. 画出信息层级和区块顺序
3. 使用现有 token 和基础组件搭建骨架
4. 补齐加载、空态、错误态
5. 接入接口和状态管理
6. 最后再做细节视觉打磨

禁止做法：

- 先堆功能，再补视觉统一
- 一个页面同时试验多种风格
- 业务逻辑和样式 token 混写在一起
- 忽略空态和错误态

---

## 11. 当前项目的落地建议

结合仓库现状，建议优先优化以下方向：

- 将首页、文章库、生词本中的重复卡片抽为共享组件
- 补充统一的空态、错误态、骨架屏组件
- 将圆角、边框、卡片背景进一步统一到 token
- 为阅读详情页增加更稳定的正文宽度控制
- 为“我的”页面建立更清晰的统计卡片视觉层级
- 后台页面与普通用户页面适当拉开语气和密度，突出管理属性

---

## 12. 验收标准

一个符合要求的前端界面，应同时满足以下标准：

- 第一眼观感干净、克制、不杂乱
- 用户能在 3 秒内理解页面主任务
- 主要按钮明显，次要按钮不抢焦点
- 标题、正文、辅助信息层级清晰
- 加载、空态、错误态完整
- 同类页面风格一致，组件复用明确
- 在桌面端宽屏下仍然保持良好阅读体验

---

## 13. 结论

EnglishAPP 前端界面应坚持“浅色、简洁、稳定、易读”的方向，避免为了“设计感”而牺牲易用性。对于当前项目，最重要的不是堆叠复杂视觉效果，而是建立统一的主题 token、稳定的卡片式布局、明确的页面层级和一致的交互反馈。

后续若继续扩展页面，建议始终基于本规范推进，以保证产品在新增功能时仍然保持统一、自然、符合大众审美的界面体验。
